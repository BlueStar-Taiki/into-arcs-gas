# 星空撮影イベント申し込み データ設計

## 基本方針

- 運用タイムゾーンは `Asia/Tokyo` とする。
- `フォームの回答 1` のフォーム所有列は生データとして編集しない。GASは末尾に追加した開催枠メタデータ列だけを補完する。
- 業務上のステータスや通知結果は `申込管理` に保持する。
- 列は位置ではなくヘッダー名で解決する。
- Discord Webhook URL はシートに保存せず、Script Properties から取得する。

## フォームの回答 1

| ヘッダー | 用途 |
|---|---|
| タイムスタンプ | フォーム受付日時 |
| お名前 | 申込者名 |
| 参加人数 | 参加者数 |
| 緊急連絡先電話番号 | 当日の緊急連絡先 |
| メールアドレス | 確認メール送信先 |
| 申し込み日時 | フォームで選択した開催枠候補 |
| タイトル | GASが補完する開催枠タイトル |
| 開催枠キー | GASが補完する開催枠照合キー |

フォーム質問として `申し込み日時` が必要である。setupはフォーム所有の列を移動せず、GAS補完列の `タイトル` と `開催枠キー` だけを末尾追加する。フォーム送信時にこの2列へ解決済み開催枠情報を書き込む。

## 申込管理

| ヘッダー | 初期値・用途 |
|---|---|
| 申込ID | `STAR-yyyyMMdd-HHmmss-XXXXXXXX` |
| 受付日時 | フォームのタイムスタンプ |
| お名前 | フォーム回答 |
| 参加人数 | フォーム回答 |
| 緊急連絡先電話番号 | フォーム回答 |
| メールアドレス | フォーム回答 |
| ステータス | 定員判定による `参加` / `キャンセル待ち` / `お断り` |
| メール送信状況 | `未送信` → `送信済み` / `送信エラー` |
| 内部メモ | 運用担当者用 |
| 最終更新日時 | 作成・通知状態更新時刻 |
| 申し込み日時 | 選択された開催枠の日時 |
| タイトル | 開催枠タイトル |
| 開催枠キー | JST日時とタイトルから生成する照合キー |

新規受付フローのプルダウン値:

- ステータス: `参加`, `キャンセル待ち`, `キャンセル`, `お断り`
- メール送信状況: `未送信`, `送信済み`, `送信エラー`

setupは申込管理の `Discord通知状況`、`カレンダー登録状況`、`イベントID` をヘッダー完全一致で削除する。他の列は削除しない。3項目とも開催日管理を正とする。

## 設定

| キー | 初期値 | 用途 |
|---|---|---|
| EVENT_NAME | 星空撮影イベント | メール・通知に表示するイベント名 |
| CONTACT_NAME | INTO-ARCS | メール送信者表示名・署名 |
| REPLY_TO_EMAIL | 空欄 | メール返信先。空欄なら指定しない |
| DISCORD_MENTION | 空欄 | Discord本文の任意メンション |
| CAPACITY_OVERBOOK_ALLOWANCE | 2 | 新規申込を参加扱いにできる定員超過の許容人数 |
| LOW_REMAINING_THRESHOLD | 4 | `残りわずか` 表示へ切り替える残席数 |

`DISCORD_MENTION` は Discord の意図しないメンションを防ぐため、Webhook payload の `allowed_mentions.parse` を空にしている。実際に通知を飛ばすメンションが必要な場合は、運用方針に合わせてコード側の許可対象も明示的に変更する。

## 開催日管理

開催枠のマスタとして、次の24列を持つ。

`申し込み日時`, `タイトル`, `定員`, `最小催行人数`, `キャンセル待ち上限`, `一人当たりの料金`, `受付開始時間`, `開催メール`, `募集状況`, `実施状況`, `最終参加人数`, `ガイド謝金`, `担当`, `参加`, `キャンセル待ち`, `キャンセル`, `お断り`, `申し込み人数`, `Discord通知状況`, `カレンダー登録状況`, `イベントID`, `内部メモ`, `勤怠登録状況`, `勤怠登録キー`

- 募集状況: `募集中`, `募集終了`, `キャンセル`
- 実施状況: `雨天中止`, `人数不足で中止`, `開催決定`, `開催済み`
- 開催メール: `未送信`, `送信済み`, `送信エラー`
- Discord通知状況: `未通知`, `最小催行人数到達通知済み`, `キャンセル待ち発生通知済み`, `両方通知済み`, `通知エラー`
- カレンダー登録状況: `未登録`, `登録済み`, `登録エラー`, `削除済み`

申込管理を正としてステータス別人数を集計する。申し込み人数は4ステータスの累計人数である。勤怠連携は未実装である。

- イベントID: Googleカレンダー登録後のイベントID。空欄行だけが登録対象
- カレンダー登録状況: 登録成功・失敗の開催枠単位状態

カレンダーイベントにはタイトル、開始日時、開始2時間後の終了日時だけを設定する。説明・場所・定員・参加人数・キャンセル待ち人数は設定しない。

募集状況が `キャンセル` でイベントIDがある場合はカレンダーイベントを削除し、イベントIDを空欄、カレンダー登録状況を `削除済み` にする。キャンセル行は新規登録しない。

Discord通知は開催枠ごとの最小催行人数到達時と、キャンセル待ち人数が0人から1人以上になった時に送る。通知状態は開催日管理で保持し、成功済み種別は再送しない。雨天中止、人数不足で中止、開催済みには送信しない。

既存データ保護のため、新しい3列はsetupで末尾追加する。イベントIDは既存列を維持し、重複作成しない。内部メモは開催枠単位のメモであり、申込管理の申込単位メモとは別物である。

## メールテンプレート

| ヘッダー | 用途 |
|---|---|
| キー | テンプレート識別子 |
| 値 | 件名または本文 |
| 説明 | 運用者向け説明 |

初期テンプレートは参加受付、キャンセル待ち受付、お断り、開催決定、雨天中止、人数不足中止、開催済みの件名・本文を持つ。setup再実行時は既存キーを上書きしない。第2段階では申込時の3種類を読み込み、差し込み後に送信する。

## ログ

| ヘッダー | 用途 |
|---|---|
| 日時 | ログ記録日時 |
| レベル | `INFO`, `WARN`, `ERROR` |
| 処理名 | 発生元の関数・処理 |
| 申込ID | 関連する申込ID |
| メッセージ | 要約 |
| 詳細 | スタックトレースなど |

## 個人情報

緊急連絡先とメールアドレスは申込管理に必要な範囲で転記する。Discord通知には緊急連絡先とメールアドレスを含めない。スプレッドシート、GASプロジェクト、Discordチャンネルの共有範囲は最小限にする。
