# 星空撮影イベント申し込み データ設計

## 基本方針

- 運用タイムゾーンは `Asia/Tokyo` とする。
- `フォームの回答 1` のフォーム所有列は生データとして編集しない。GASは末尾に追加した開催枠メタデータ列だけを補完する。
- 業務上のステータスや通知結果は `申込管理` に保持する。
- 列は位置ではなくヘッダー名で解決する。
- Discord Webhook URL はシートに保存せず、Script Properties から取得する。

## フォームの回答 1

| ヘッダー | 用途 |
|---|---|
| タイムスタンプ | フォーム受付日時 |
| お名前 | 申込者名 |
| 参加人数 | 参加者数 |
| 緊急連絡先電話番号 | 当日の緊急連絡先 |
| メールアドレス | 確認メール送信先 |
| 申し込み日時 | フォームで選択した開催枠候補 |
| タイトル | GASが補完する開催枠タイトル |
| 開催枠キー | GASが補完する開催枠照合キー |

フォーム質問として `申し込み日時` が必要である。setupはフォーム所有の列を移動せず、GAS補完列の `タイトル` と `開催枠キー` だけを末尾追加する。フォーム送信時にこの2列へ解決済み開催枠情報を書き込む。

## 申込管理

| ヘッダー | 初期値・用途 |
|---|---|
| 申込ID | `STAR-yyyyMMdd-HHmmss-XXXXXXXX` |
| 受付日時 | フォームのタイムスタンプ |
| お名前 | フォーム回答 |
| 参加人数 | フォーム回答 |
| 緊急連絡先電話番号 | フォーム回答 |
| メールアドレス | フォーム回答 |
| ステータス | 定員判定による `参加` / `キャンセル待ち` / `お断り` |
| メール送信状況 | `未送信` → `送信済み` / `送信エラー` |
| お支払い | 新規申込時は `未払い`。入金確認後に `お支払い済み` へ変更する |
| 内部メモ | 運用担当者用 |
| 最終更新日時 | 作成・通知状態更新時刻 |
| 申し込み日時 | 選択された開催枠の日時 |
| タイトル | 開催枠タイトル |
| 開催枠キー | JST日時とタイトルから生成する照合キー |

新規受付フローのプルダウン値:

- ステータス: `参加`, `キャンセル待ち`, `キャンセル`, `お断り`
- メール送信状況: `未送信`, `送信済み`, `送信エラー`
- お支払い: `未払い`, `お支払い済み`

setupは申込管理の `Discord通知状況`、`カレンダー登録状況`、`イベントID` をヘッダー完全一致で削除する。他の列は削除しない。3項目とも開催日管理を正とする。

## 担当

ガイド担当者のマスタとして、次の5列を持つ。

`担当者名`, `メールアドレス`, `出勤簿キー`, `出勤簿登録対象`, `メモ`

- 担当者名: 開催日管理の `担当` と完全一致で照合するキー。
- メールアドレス: ガイド向け実施状況メールの送信先。空欄の場合は送信せずログに記録する。
- 出勤簿キー: 将来の出勤簿連携用。今回のメール送信では使用しない。
- 出勤簿登録対象: `対象`, `対象外` のプルダウン。今回のメール送信では必須条件にしない。
- メモ: 任意メモ。GAS処理では基本的に使用しない。

setupは既存データを保持し、不足ヘッダーだけを末尾追加する。

## 開催日管理

開催枠のマスタとして、次の29列を持つ。

`申し込み日時`, `タイトル`, `定員`, `最小催行人数`, `キャンセル待ち上限`, `一人当たりの料金`, `受付開始時間`, `開催メール`, `募集状況`, `実施状況`, `最終参加人数`, `ガイド謝金`, `担当`, `参加`, `キャンセル待ち`, `キャンセル`, `お断り`, `申し込み人数`, `Discord通知状況`, `カレンダー登録状況`, `イベントID`, `内部メモ`, `勤怠登録状況`, `勤怠登録キー`, `最小催行通知`, `キャンセル待ち通知`, `5日前通知`, `参加者名簿URL`, `前回実施状況`

- 募集状況: `募集中`, `募集終了`, `キャンセル`
- 実施状況: `雨天中止`, `人数不足で中止`, `開催決定`, `開催済み`
- 開催メール: `未送信`, `送信済み`, `送信エラー`
- Discord通知状況: `未通知`, `最小催行人数到達通知済み`, `キャンセル待ち発生通知済み`, `両方通知済み`, `通知エラー`
- 最小催行通知: `未通知`, `通知済み`, `通知エラー`
- キャンセル待ち通知: `未通知`, `通知済み`, `通知エラー`
- 5日前通知: `未通知`, `通知済み`, `通知エラー`
- カレンダー登録状況: `未登録`, `登録済み`, `登録エラー`, `削除済み`
- 参加者名簿URL: `開催決定` 時に作成したPDFの共有URL
- 前回実施状況: 確認ダイアログで「いいえ」を選んだ場合や送信エラー時に戻す内部値。setupで非表示にする

申込管理を正としてステータス別人数を集計する。申し込み人数は4ステータスの累計人数である。`担当` は担当シートの `担当者名` と完全一致で照合し、単一担当は担当シートの名前をプルダウンから選択できる。複数担当は `,`、`、`、改行、`/` で区切って手入力できるよう、入力規則は警告のみとする。勤怠連携は未実装である。

- イベントID: Googleカレンダー登録後のイベントID。空欄行だけが登録対象
- カレンダー登録状況: 登録成功・失敗の開催枠単位状態

カレンダーイベントにはタイトル、開始日時、開始2時間後の終了日時だけを設定する。説明・場所・定員・参加人数・キャンセル待ち人数は設定しない。

募集状況が `キャンセル` でイベントIDがある場合はカレンダーイベントを削除し、イベントIDを空欄、カレンダー登録状況を `削除済み` にする。キャンセル行は新規登録しない。

Discord通知は開催枠ごとの最小催行人数到達時、フォーム候補が `【キャンセル待ち】` になる状態への到達時、開催日の5日前に送る。3つの個別通知列を正とし、Discord通知状況は既存2通知の総合表示とする。成功済み種別は再送しない。雨天中止、人数不足で中止、開催済みには送信しない。

既存データ保護のため、新しい列はsetupで末尾追加する。イベントIDは既存列を維持し、重複作成しない。内部メモは開催枠単位のメモであり、申込管理の申込単位メモとは別物である。

## 参加者名簿

`参加者名簿` シートはPDF出力用の雛形である。開催日管理の `実施状況` を `開催決定` にして確認ダイアログで「はい」を選んだ時、同じ開催枠の `参加` 申込から `お名前` と `参加人数` を書き込み、PDFとして設定 `PARTICIPANT_ROSTER_FOLDER_ID` のDriveフォルダに保存する。PDFはリンクを知っている全員が閲覧できる共有設定にし、共有URLを開催日管理の `参加者名簿URL` に保存する。

## メールテンプレート

| ヘッダー | 用途 |
|---|---|
| キー | テンプレート識別子 |
| 値 | 件名または本文 |
| 説明 | 運用者向け説明 |

初期テンプレートは参加受付、キャンセル待ち受付、お断り、お支払い確認、参加者向けの開催決定・雨天中止・人数不足中止・開催済み、ガイド向けの開催決定・雨天中止・人数不足中止・開催済みの件名・本文を持つ。setup再実行時は既存キーを上書きしない。参加受付メールにはクレジットカード事前払いの案内と `PAYMENT_LINK` を記載する。お支払い確認メールは `payment_confirmed_subject/body` を使う。参加者向け実施状況メールは `event_confirmed_*`, `event_rain_cancel_*`, `event_insufficient_cancel_*`, `event_completed_*` を使う。ガイド向け実施状況メールは `guide_event_confirmed_*`, `guide_event_rain_cancel_*`, `guide_event_insufficient_cancel_*`, `guide_event_completed_*` を使う。差し込みには `{{実施状況}}` も利用でき、未解決差し込みが残る場合は送信しない。

ガイド向け差し込み:

- `{{担当者名}}`
- `{{申し込み日時}}`
- `{{タイトル}}`
- `{{受付開始時間}}`
- `{{実施状況}}`
- `{{開催可否}}`
- `{{開催できない理由}}`
- `{{参加人数}}`
- `{{キャンセル待ち人数}}`
- `{{定員}}`
- `{{最小催行人数}}`
- `{{一人当たりの料金}}`
- `{{主催者名}}`

## 個人情報

緊急連絡先とメールアドレスは申込管理に必要な範囲で転記する。Discord通知には緊急連絡先とメールアドレスを含めない。スプレッドシート、GASプロジェクト、Discordチャンネルの共有範囲は最小限にする。
