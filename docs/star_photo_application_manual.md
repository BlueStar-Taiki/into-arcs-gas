# 星空撮影イベント申し込み GAS 運用マニュアル

## 対象

- スプレッドシート: `星空撮影イベント申し込み`
- スプレッドシート ID: `1f7s_LQfI6GrVy5O99pTOrbk2XE1r3m4p5b5okIA0M-o`
- フォーム: `https://forms.gle/yw1MrgfEw6JrzVrp7`
- GAS ソース: `gas/star-photo-application/src`

## 導入前の確認

1. Google フォームの回答先が対象スプレッドシートになっていることを確認する。
2. 回答タブ名が `フォームの回答 1` であることを確認する。
3. 回答タブのヘッダーがスキーマ記載の5項目と一致することを確認する。
4. このGASを対象スプレッドシートのコンテナバインド型プロジェクトとして配置する。

`.clasp.json`、Webhook URL、カレンダーID、認証情報、フォーム回答などの個人情報はリポジトリへコミットしない。

## Script Properties

GASエディタの「プロジェクトの設定」→「スクリプト プロパティ」で次を設定する。

| プロパティ | 必須 | 値 |
|---|---|---|
| `DISCORD_WEBHOOK_URL` | 必須 | 対象チャンネルのWebhook URL |
| `APPLICATION_FORM_ID` | 必須 | フォーム編集URLの `/d/` と `/edit` の間にあるID |
| `CALENDAR_ID` | カレンダー登録時に必須 | 登録先GoogleカレンダーのID |

Webhook URLそのものを `設定` シート、ソースコード、ログへ記載しない。GASは値を保存・表示せず、Script Propertiesにキーと空でない値が存在するかだけを確認する。

`CALENDAR_ID` はGoogleカレンダーの「設定と共有」→「カレンダーの統合」で確認し、Script Propertiesだけに保存する。未設定でもsetupは成功するが、`checkApplicationFormSetup()` は警告を返し、カレンダー登録実行時はエラーになる。

## 初回セットアップ

1. GASエディタへ `src` 配下のJavaScriptファイルを追加する。
2. プロジェクトのタイムゾーンも `Asia/Tokyo` に設定する。
3. Googleフォームにタイトルが正確に `申し込み日時` のプルダウン質問を1件作成する。仮の候補を1件入れて保存し、回答シートに同名ヘッダーが追加されたことを確認する。反映されない場合は一度テスト回答を送信して確認する。
4. 「プロジェクトの設定」→「スクリプト プロパティ」で `DISCORD_WEBHOOK_URL` と `APPLICATION_FORM_ID`、カレンダー登録を使う場合は `CALENDAR_ID` を設定する。
5. GASエディタ上部の関数選択欄で `setupApplicationFormSystem` を選び、「実行」を1回押す。
6. 初回に表示されるGoogleの権限承認画面で、内容を確認して許可する。
7. `申込管理`、`開催日管理`、`メールテンプレート`、`設定`、`ログ` の構成と、ログに記録されたセットアップ結果を確認する。
   - 申込管理から `Discord通知状況`、`カレンダー登録状況`、`イベントID` が削除されている
   - 申込管理のその他の列が保持されている
   - ログに削除した3列が記録されている
8. `設定` シートの `EVENT_NAME`、`CONTACT_NAME`、必要なら `REPLY_TO_EMAIL` を編集する。
9. 開催日管理に開催枠を入力し、募集状況を空欄または `募集中` にする。
10. `updateApplicationFormChoices()` を実行する。この関数内で人数集計も更新される。
11. Apps Script左側の「トリガー」で、次のトリガーが1件あることを確認する。
   - 実行する関数: `onFormSubmit`
   - イベントのソース: スプレッドシートから
   - イベントの種類: フォーム送信時
12. スプレッドシートを再読み込みし、`INTO-ARCS管理` メニューが表示されることを確認する。

初回権限承認はGoogleの仕様上自動化できないため、人間による操作が必要である。シートの作成・更新、Googleフォームの候補編集、Script Propertiesの参照、インストール型トリガーの作成、メール送信、DiscordへのHTTP通信に必要な権限を許可する。

`setupApplicationFormSystem()` は既存の `setupApplicationFormSheet()` を呼び出した後、必須Script Propertiesを点検し、対象スプレッドシート用のトリガーがなければ作成する。既に同じトリガーがある場合は重複作成しない。`onFormSubmit` はGASエディタから直接実行してはならない。

セットアップ後の再点検には `checkApplicationFormSetup()` を実行する。これはシートやトリガーを変更せず、点検結果をログシートへ追記する。

## 開催日管理拡張の段階導入

`setupApplicationFormSheet()` は開催日管理とメールテンプレートも再現する。開催日管理では、申し込み日時、タイトル、定員、最小催行人数、キャンセル待ち上限、一人当たりの料金、受付開始時間を人間が入力する。

第2段階では開催枠の集計、申込時の参加・キャンセル待ち・お断り判定、フォーム候補同期、ステータス別メールを有効化している。カレンダー登録と開催枠単位のDiscord通知も実装済みで、実施状況メールは未実装である。申込ごとのDiscord通知は停止している。

メールテンプレートの初期値はsetupで追加され、既存キーの値は上書きされない。申込メールでは `{{お名前}}` などを置換し、未解決の差し込みが残る場合は送信せずログへ記録する。

## 開催枠単位Discord通知

各開催枠で次のタイミングに一度だけ送信する。

- 参加人数が最小催行人数へ初めて到達した時
- キャンセル待ち人数が0人から1人以上になった時

フォーム申込後に自動判定するほか、`INTO-ARCS管理` → `開催枠Discord通知を確認・送信` から手動確認できる。雨天中止、人数不足で中止、開催済みには送信しない。

Discord通知状況:

- `未通知`
- `最小催行人数到達通知済み`
- `キャンセル待ち発生通知済み`
- `両方通知済み`
- `通知エラー`

成功済み通知は再送しない。片方だけ成功した後にもう片方が失敗した場合は、成功済み状態を保持し、次回は未送信側だけを再試行する。

## Googleカレンダー登録

1. Script Propertiesへ `CALENDAR_ID` を設定する。
2. 開催日管理の対象行に申し込み日時とタイトルを入力する。
3. イベントIDが空欄であることを確認する。
4. スプレッドシートの `INTO-ARCS管理` → `開催枠をカレンダー登録` を実行する。
5. 開催日管理のイベントIDとカレンダー登録状況を確認する。
6. エラー時は登録状況が `登録エラー` になった行とログシートを確認する。

登録対象は、募集状況がキャンセル以外で、イベントIDが空欄、申し込み日時あり、タイトルありの行だけである。作成するイベントはタイトルと開始・終了日時だけを持ち、終了は開始の2時間後となる。説明と場所は設定せず、定員や参加状況も書き込まない。

開催枠を中止する場合は、開催日管理の募集状況を `キャンセル` にして、同じメニューを実行する。イベントIDがある行はカレンダーから削除され、イベントIDが空欄、カレンダー登録状況が `削除済み` になる。キャンセル状態の行は新規登録されない。

## 定員・フォーム表示設定

設定シートで次の値を変更できる。setup再実行時は不足キーだけが追加され、既存値は上書きされない。

| キー | 初期値 | 意味 |
|---|---:|---|
| `CAPACITY_OVERBOOK_ALLOWANCE` | 2 | 既存参加人数が定員を超えた場合の内部許容・警告人数 |
| `LOW_REMAINING_THRESHOLD` | 4 | フォームで `【残りわずか】` と表示する残席数の上限 |

どちらも0以上の整数で入力する。空欄、負数、小数、数値以外の場合、システムはデフォルト値へ戻さず、申込判定またはフォーム候補更新をエラーにする。

例:

- 定員20、現在参加18、2名申込: 合計20なので `参加`
- 定員20、現在参加19、2名申込: 合計21なので `キャンセル待ち`
- 定員20、現在参加20、1名申込: 定員超過なので `キャンセル待ち`
- 残席5: `【残り5人】`
- 残席4〜1: `【残りわずか】`
- 残席0で待機可能: `【キャンセル待ち】`

`CAPACITY_OVERBOOK_ALLOWANCE` は通常の新規申込を参加扱いに広げる設定ではない。管理者の手動変更などで、既に参加扱いの人数が定員を超えた場合の将来の警告管理に使用する。募集終了は `参加 + キャンセル待ち >= 定員 + キャンセル待ち上限` で判定する。

## 動作確認

1. テスト用のメールアドレスでGoogleフォームから1件送信する。
2. `フォームの回答 1` に元回答が追加されていることを確認する。
3. `申込管理` に同内容と申込IDが追加されていることを確認する。
4. 初期状態が定員判定に応じた `参加` / `キャンセル待ち` / `お断り` であることを確認する。
5. 確認メールを受信し、メール送信状況が `送信済み` になることを確認する。
6. `ログ` にフォーム処理完了ログがあることを確認する。
7. `申込管理` の対象データ行を1行選択し、管理メニューからメール再送を確認する。
8. テスト申込に個人情報を使った場合は、確認後に運用ルールに従って削除する。
9. 定員内の申込が `参加`、全員分の残席がない申込が `キャンセル待ち`、待機上限も超える申込が `お断り` になることを確認する。
10. 開催日管理の5集計列と募集状況、フォームの残席表示が更新されることを確認する。
11. 開催日管理にDiscord通知状況、カレンダー登録状況、イベントID、内部メモが一度ずつ存在することを確認する。
12. 設定値を一時的に不正値にした場合、分かりやすいエラーになり、勝手に初期値へ戻らないことを確認する。
13. CALENDAR_ID未設定でsetupが停止せず、点検だけが警告になることを確認する。
14. テスト用開催枠を登録し、タイトル・開始日時・2時間後の終了日時だけがカレンダーへ登録されることを確認する。
15. 登録済み枠の募集状況を `キャンセル` にして同期し、カレンダーから削除、イベントID空欄、登録状況 `削除済み` になることを確認する。
16. 最小催行人数到達時とキャンセル待ち初回発生時に、それぞれ一度だけDiscord通知され、最終的に `両方通知済み` になることを確認する。

## エラー時

- Apps Script左側の「実行数」で、失敗した関数と例外内容を確認する。
- `ログ` シートで `setupApplicationFormSystem`、`checkApplicationFormSetup`、`onFormSubmit` の記録を確認する。
- 「プロジェクトの設定」で `DISCORD_WEBHOOK_URL` と `APPLICATION_FORM_ID` が存在し、値が空でないことを確認する。値をログやシートへ転記しない。
- カレンダー登録エラー時は `CALENDAR_ID`、開催日管理の登録状況、ログシートを確認する。カレンダーID自体はログへ転記しない。
- Discord通知エラー時は開催日管理のDiscord通知状況とログシートを確認し、管理メニューから再試行する。Webhook URL自体はログへ転記しない。
- Apps Script左側の「トリガー」で `onFormSubmit` が1件だけ存在することを確認する。
- メールだけが失敗しても、申込管理への転記は維持される。
- メール送信状況が `送信エラー` になり、`ログ` に詳細が残る。
- 原因を修正後、申込管理の対象行を1行選択し、管理メニューから再送する。
- フォーム項目名を変更した場合は `config.js` のヘッダー定義も同時に更新する。

## 注意点

- `setupApplicationFormSheet()` は申込管理の廃止列 `Discord通知状況`、`カレンダー登録状況`、`イベントID` だけを削除する。他の業務列やデータは削除せず、設定値も上書きしない。
- `listApplicationFormTriggers()` は現在のプロジェクトトリガー情報を実行ログへ出力し、配列で返す。
- `removeDuplicateApplicationFormTriggers()` は対象スプレッドシートの重複トリガーを1件だけ残して削除する。削除操作なので、一覧を確認したうえで必要な場合のみ人間が明示的に実行する。
- 管理シートに想定外のヘッダーがある場合は、安全のためセットアップを中断する。
- 第2段階以前の既存申込には開催枠キーを自動補完しないため、新しい人数集計には含まれない。推測による誤紐付けを避けるためで、必要なら別途確認付き移行を行う。
- プルダウンと日時書式は初期状態で1000データ行分設定する。必要に応じて `VALIDATION_ROW_COUNT` を増やす。
- 同時送信時の行競合を避けるため、フォーム処理はスクリプトロックを使用する。
- 開催枠単位のDiscord通知状況、カレンダー登録状況、イベントIDは開催日管理を正とし、申込管理では保持しない。
- 申込管理の内部メモは申込単位、開催日管理の内部メモは開催枠単位で使用する。
- `APPLICATION_FORM_ID` は必須である。`CALENDAR_ID` はsetupには任意、カレンダー登録実行時には必須である。実値をシートやコードへ記録しない。
